================================================================================
                               DESCRIPCIÓN
================================================================================

Sistema de autenticación JWT (JSON Web Tokens) seguro, escalable y fácil de
implementar para aplicaciones multi-usuario. Incluye:

✅ Registro de usuarios con validación
✅ Login con email/password
✅ JWT con expiración corta (15 minutos)
✅ Refresh tokens (7 días) con rotación
✅ Auto-refresh automático (5 min antes de expirar)
✅ Logout con invalidación de tokens
✅ Rate limiting contra brute force
✅ Auditoría de accesos
✅ Password hashing con bcrypt (12 rounds)
✅ Middleware de protección de rutas
✅ Validación de roles (user, admin, moderador)

================================================================================
                             QUICK START (5 MIN)
================================================================================

1. Instalar dependencias:
   cd backend
   npm install

2. Crear archivo .env:
   JWT_SECRET=openssl rand -base64 32 (generar)
   REFRESH_TOKEN_SECRET=openssl rand -base64 32 (generar)
   NODE_ENV=development
   PORT=3000
   SUPABASE_URL=https://your-project.supabase.co
   SUPABASE_ANON_KEY=your-key
   ALLOWED_ORIGINS=http://localhost:3000

3. Ejecutar SQL en Supabase:
   • Copiar contenido de: backend/database/migrations/001-auth-schema.sql
   • Ejecutar en Supabase SQL Editor

4. Inicializar Express:
   const { createAuthRoutes } = require('./backend/routes/auth-routes');
   const { authMiddleware } = require('./backend/middleware/auth-middleware');

   app.use('/api/auth', createAuthRoutes(supabaseClient));
   app.get('/api/protected', authMiddleware(supabaseClient), handler);

5. Integrar frontend:
   <script src="auth/auth-client.js"></script>
   const auth = new AuthClient('http://localhost:3000/api');

6. Probar:
   curl -X POST http://localhost:3000/api/auth/login \
     -H "Content-Type: application/json" \
     -d '{"email":"test@example.com","password":"Test1234"}'

================================================================================
                          FLUJO DE AUTENTICACIÓN
================================================================================

REGISTRO:
  Frontend → POST /auth/register {email,password,nombre,empresa}
           ↓ (validar datos)
  Backend  → Hash password (bcrypt), crear usuario, generar tokens
           ↓ (guardar refresh token en BD)
  Frontend ← 201 {token, refreshToken, usuario_id, etc.}
           ↓ (guardar en localStorage)
           ↓ (iniciar auto-refresh)
           ↓ (redirigir a dashboard)

LOGIN:
  Frontend → POST /auth/login {email, password}
           ↓ (validar datos)
  Backend  → Buscar usuario, comparar password con bcrypt
           ↓ (si válido: generar tokens, guardar refresh_token en BD)
  Frontend ← 200 {token, refreshToken, usuario_id, etc.}
           ↓ (guardar en localStorage)
           ↓ (iniciar auto-refresh en 10 minutos)
           ↓ (redirigir a dashboard)

AUTO-REFRESH (cada 5 min antes de expiración):
  Frontend → POST /auth/refresh {refreshToken}
           ↓ (5 minutos antes de expiración)
  Backend  → Verificar refresh token, invalidar antiguo
           ↓ (Token Rotation: generar nuevo par de tokens)
  Frontend ← 200 {newToken, newRefreshToken}
           ↓ (actualizar localStorage)
           ↓ (reiniciar timer de auto-refresh)

LOGOUT:
  Frontend → POST /auth/logout
           ↓ (enviar token en Authorization header)
  Backend  → Agregar a blacklist, invalidar refresh tokens
           ↓ (en BD: todos los refresh tokens = inactivo)
  Frontend ← 200 {mensaje}
           ↓ (limpiar localStorage)
           ↓ (detener auto-refresh)
           ↓ (redirigir a login)

ACCEDER A RUTA PROTEGIDA:
  Frontend → GET /api/inventario
           ↓ (header: Authorization: Bearer {token})
  Middleware → Extraer token de header
            → Verificar signature JWT
            → Verificar no esté en blacklist
            → Verificar no expirado
            ↓ (si válido: req.usuario = {usuario_id, email, rol})
  Route    → Acceso a req.usuario
            ← 200 {datos}
  Frontend ← Renderizar datos

================================================================================
                        SEGURIDAD IMPLEMENTADA
================================================================================

✅ Password Hashing:
   • Algoritmo: bcryptjs
   • Salt rounds: 12 (muy seguro)
   • Nunca almacenar password plano

✅ JWT Signing:
   • Algoritmo: HS256 (HMAC-SHA256)
   • Secret: Variable de entorno (no en código)
   • Verificación en cada request

✅ Token Rotation:
   • Refresh token se invalida después de usar
   • Nuevo refresh token generado automáticamente
   • Previene token reuse attacks

✅ Token Blacklist:
   • Logout invalida tokens
   • En memoria (production: usar Redis)
   • TTL de 15 minutos

✅ Rate Limiting:
   • Max 5 intentos de login / 15 minutos
   • Por email o IP
   • Previene brute force

✅ Auditoría:
   • Logging de login/logout
   • Timestamp de último login
   • Tabla audit_log para eventos

✅ Validación:
   • Email format validation
   • Password strength: min 8 chars, mayúscula, número
   • Input sanitization
   • Express-validator en todas las rutas

✅ CORS:
   • Whitelist de orígenes permitidos
   • Credenciales solo con HTTPS

✅ Headers de Seguridad:
   • HSTS (Force HTTPS)
   • CSP (Content Security Policy)
   • X-Content-Type-Options
   • X-Frame-Options
   • (Implementar con helmet.js)

================================================================================
                            PRUEBAS UNITARIAS
================================================================================

Tests recomendados para CI/CD:

1. Registro:
   ✓ Email válido
   ✓ Password fuerte
   ✓ Usuario no existe
   ✓ Rechazar email existente
   ✓ Rechazar password débil

2. Login:
   ✓ Credenciales válidas
   ✓ Email incorrecto
   ✓ Password incorrecto
   ✓ Cuenta inactiva
   ✓ Rate limiting después de 5 intentos

3. Token:
   ✓ Verificar token válido
   ✓ Rechazar token expirado
   ✓ Rechazar token modificado
   ✓ Verificar payload correcto

4. Refresh:
   ✓ Refrescar con refresh token válido
   ✓ Rechazar refresh token expirado
   ✓ Invalidar token anterior (rotation)
   ✓ Renovar timer de auto-refresh

5. Logout:
   ✓ Invalidar token
   ✓ Agregar a blacklist
   ✓ No aceptar token invalidado

6. Middleware:
   ✓ Rechazar sin Authorization header
   ✓ Rechazar con formato incorrecto
   ✓ Rechazar token inválido
   ✓ Permitir token válido

7. Rate Limiting:
   ✓ Permitir < 5 intentos
   ✓ Bloquear en 5to intento
   ✓ Permitir después de 15 minutos

================================================================================
                       MONITOREO EN PRODUCCIÓN
================================================================================

Métricas importantes:
  • Login success/failure ratio
  • Token refresh rate
  • Failed login attempts
  • Unique IPs por usuario
  • Logout ratio (debe ser 80-90%)
  • Rate limit blocks
  • Average response time

Alertas:
  • Brute force detection (> 5 fallos en 15 min)
  • Anomalía de IP
  • Token refresh anormal
  • Base de datos desconexión
